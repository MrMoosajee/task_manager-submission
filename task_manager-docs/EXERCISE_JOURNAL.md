Technical Journal: Task Manager ExplorationExercise Part 1: Task Creation and Status UpdatesCore Componentscli.py: The entry point that parses user commands (e.g., create or status) and handles terminal output.  app.TaskManager: The service layer that validates inputs (like date formats) and orchestrates logic between the UI and storage.  models.Task: The data blueprint that defines the structure of a task, including its unique UUID and timestamps.  storage.TaskStorage: The persistence layer responsible for the custom serialization of objects into the tasks.json file.  Execution Flow (Creation)User Input: User executes python cli.py create "New Task".  Parsing: argparse in cli.py captures the title, priority, and due date.  Logic Execution: TaskManager.create_task is called, which converts the raw priority integer into a TaskPriority Enum and parses the date string into a datetime object.  Instantiation: A new Task object is created with a unique ID.  Persistence: TaskStorage.add_task appends the object to the internal dictionary and triggers save(), which uses TaskEncoder to write to the JSON file.  Exercise Part 2: The Task Prioritization SystemInitial vs. Final UnderstandingInitial: I assumed priority was just a simple integer (1–4) stored in a list.Discovered: The system uses a Strongly Typed Enum (TaskPriority). This is crucial because it prevents invalid priority levels from ever reaching the storage layer. The TaskEncoder specifically handles converting these Enums into JSON-friendly integers and back again.  Guided InsightsInput Validation: The CLI uses choices=[1, 2, 3, 4] in argparse, providing the first layer of defense.  Mapping: The format_task function in cli.py maps these priority levels to visual "!" symbols, translating raw data into a user-friendly UI. Timestamps: The completed_at attribute is updated from None to the current datetime.now(), and updated_at is refreshed.  Potential Points of FailureInvalid ID: If the user provides a partial or incorrect UUID, get_task returns None, and the update fails gracefully with an error message.  Serialization Error: If a new attribute is added to Task but not handled in TaskEncoder, the save() method might fail to persist the completion state.  Exercise Part 4: Final Presentation & ReflectionApplication Architecture OverviewThe Task Manager follows a Layered Architecture. By separating the Domain Model (the data) from Storage (the file handling) and App Logic (the "brain"), the codebase remains modular. This allowed me to understand how to add a feature (like "Abandoned" status) just by updating the Enum in models.py and the logic in app.py without breaking the UI.  Key Feature WorkflowsCreation: Leverages uuid for unique tracking and json for persistence.  Prioritization: Uses Enums to ensure data integrity across the stack.  Completion: Updates specific state flags (status, completed_at) and triggers a full rewrite of the JSON file to ensure the disk matches the memory.  ReflectionThe most challenging aspect was the Custom JSON Serialization in storage.py. Standard JSON libraries can't handle Python's datetime or Enum objects directly. Understanding how the TaskEncoder and TaskDecoder "hand-shake" to translate these objects into strings was a major breakthrough in my understanding of the project's data lifecycle.
